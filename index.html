Chapter 01

System Overview & Vision

RoadSense is a mobile-first web application that uses a smartphone's built-in sensors — accelerometer, gyroscope, magnetometer, and GPS — to detect and classify road surface conditions in real time. It adapts to each individual rider's behavior, vehicle type, and riding style over time, producing increasingly accurate road condition labels without manual intervention.

The system consists of three tightly integrated components working together:

Frontend Web App

A single HTML file. No build step, no install. Runs fully in the phone browser. Collects sensor data, runs the adaptive engine locally, shows a live GPS map, and lets riders report obstructions.

Backend API + Database

A Node.js + Express server that receives uploaded ride data, stores it in SQLite with metadata, and saves labeled CSV files to disk for future ML training.

Admin Dashboard

A password-protected web dashboard that shows all collected rides, label distribution charts, per-ride download links, and a live Leaflet map of all road condition data overlaid across all riders — a growing ground truth city road quality map.

Why This Approach

Traditional road condition survey systems require specialized hardware, manual inspection teams, or government infrastructure. RoadSense turns every smartphone into a sensor node. The key insight is that raw sensor thresholds differ massively between a bicycle, a motorcycle, and a car — and between a cracked old phone and a new flagship. Hardcoded thresholds fail. The adaptive calibration engine solves this by learning the rider's specific baseline over rides, making the system universally accurate regardless of device or vehicle.

📌 Core Principle
The system does not use a pre-trained ML model. The adaptive engine is a statistical calibration system — it learns thresholds per rider using rolling statistics (mean, standard deviation) on their own data. The labeled CSVs it produces are the training data for a future ML model.

Chapter 03

Frontend — Adaptive Web App

The frontend is a single .html file with no dependencies except Leaflet.js (loaded from CDN). It works fully offline once opened — the only network call is the optional upload to your server at the end of a ride.

Four Screens

Screen 1 — Home / Profile

Displays the rider's profile loaded from localStorage: calibration confidence score (0–100%), number of rides completed, number of corrections made, and current threshold values. A "Start Ride" button begins data collection.

Screen 2 — Live Ride

Split into two panels. The top panel shows live sensor values (ax, ay, az, gx, gy, gz, orientation α/β/γ), a scrolling real-time magnitude waveform for the last 5 seconds, and a large color-coded road condition badge that updates every 0.3 seconds showing the current window's label.

The bottom panel is a Leaflet map centered on the rider's GPS position. As the ride progresses, a colored polyline is drawn segment by segment reflecting the road condition at each GPS coordinate. Event markers (emoji icons) are dropped at notable events.

A large "⊘ OBSTRUCTION" press-and-hold button sits prominently at the bottom. Holding it marks the duration, releasing triggers a sub-type selector sheet.

Screen 3 — Post-Ride Review

Shows the complete ride on the Leaflet map with the full colored route and all markers. Below the map is a horizontal scrollable timeline of all 0.3s windows rendered as colored chips. Tapping any chip opens a popup showing the sensor values for that window and the current label, with buttons to change it. Every correction feeds back into the adaptive engine.

Screen 4 — Upload & Export

Automatically POSTs the ride data to the server. Shows upload progress. Also provides a "Download CSV" button for a local copy. Displays a summary: sample count, window count, label distribution breakdown.

Chapter 04

The Adaptive Calibration Engine

This is the core intelligence of RoadSense. Instead of hardcoded magnitude thresholds, the engine maintains a statistical model of each rider's baseline (smooth road) sensor behavior, and derives all thresholds as multiples of that baseline's standard deviation.

Three Features Per Window

FeatureFormulaWhat it detects
Accel Magnitude√(ax² + ay² + az²)Overall intensity of road disturbance
Jerk|Δmagnitude / Δtime|Sharpness of event — distinguishes sharp potholes from gradual humps
Gyro Norm√(gx² + gy² + gz²)Rotational disturbance — obstructions, severe sway, bad road

Speed Normalization

A vehicle traveling at 60 km/h over the same pothole produces a higher magnitude signal than one at 20 km/h. The engine compensates using GPS speed:

adjusted_magnitude = raw_magnitude / speed_factor speed_factor = 1.0 + (gps_speed_kmh / 60.0) // Example: 80 km/h → speed_factor = 2.33 // same pothole at 80 km/h reads as if at 34 km/h equivalent

Label Decision Logic

After speed normalization, the adjusted magnitude is compared against dynamic thresholds derived from the rider's baseline:

μ = baseline.mean // rider's smooth-road mean σ = baseline.std // rider's smooth-road std deviation T = thresholds // learned multipliers IF adj_mag < μ + T.avg × σ → good ELIF adj_mag < μ + T.hump × σ AND jerk < T.jerk → avg or hump ELIF adj_mag < μ + T.hump × σ AND jerk ≥ T.jerk → pothole ELIF adj_mag < μ + T.bad × σ → bad ELIF gyro_norm > μ_gyro + T.gyro × σ_gyro → obstruction* ELSE → bad * sensor-detected only; rider press-and-hold overrides with sub-type

Baseline Learning (Welford's Online Algorithm)

The baseline updates incrementally every window — no need to store all past samples in memory. Windows labeled good contribute to the baseline. This is key: as the rider corrects labels, more accurate good-road windows enter the baseline.

// Welford's online mean + variance update
function updateBaseline(profile, newMagnitude) {
  profile.baseline.sampleCount++;
  const delta = newMagnitude - profile.baseline.mean;
  profile.baseline.mean += delta / profile.baseline.sampleCount;
  const delta2 = newMagnitude - profile.baseline.mean;
  profile.baseline.M2 += delta * delta2;  // running sum of squares
  profile.baseline.std = Math.sqrt(
    profile.baseline.M2 / profile.baseline.sampleCount
  );
}

Correction Learning (Threshold Nudging)

When a rider corrects a label, the relevant threshold multiplier is nudged by a small learning rate. This is a simple but effective online learning mechanism:

const LEARNING_RATE = 0.05;  // 5% nudge per correction

// If rider corrects "hump" → "good": hump threshold was too low, raise it
if (predicted === 'hump' && corrected === 'good') {
  profile.thresholds.humpMultiplier *= (1 + LEARNING_RATE);
}

// If rider corrects "good" → "pothole": threshold was too high, lower it
if (predicted === 'good' && corrected === 'pothole') {
  profile.thresholds.potholeMultiplier *= (1 - LEARNING_RATE);
}

Calibration Confidence Score

A 0–100% score shown on the home screen. Gives users feedback on how well-calibrated their profile is:

confidence = min(100, (rideCount × 8) + (correctionCount × 1.5)) // 0 rides = 0% (default thresholds, unreliable) // 5 rides = 40% (getting personalized) // 10 rides = 80% (well calibrated) // 13 rides = 100% (fully calibrated)
✓ Key Insight
The engine never needs to store past sensor data. All learning happens through incrementally updated statistics. The entire rider profile is under 1KB in localStorage.
Chapter 05

Live Map & GPS Integration

The map uses Leaflet.js + OpenStreetMap tiles — completely free, no API key required. GPS data is collected via the browser's Geolocation.watchPosition() API with high accuracy mode enabled.

How the Colored Polyline Works

Rather than one continuous polyline, the map draws a new line segment for each 0.3s window. Each segment connects the GPS coordinates of consecutive windows and is colored based on that window's label:

LabelColorMap marker
good#39ff6e — bright greenNone (clean road)
avg#ffd93d — yellowNone
hump#ff9f43 — orange∧ marker
pothole#ff6b81 — pink-red🕳 marker
bad#ff4757 — red✕ marker
obstruction#a29bfe — purple⊘ marker with sub-type popup

GPS Stamping on Sensor Samples

GPS updates arrive at ~1Hz, far slower than the IMU (~100Hz). The last known GPS coordinate is carried forward and stamped on every sensor sample until a new GPS fix arrives. This means every row in the CSV has a lat/lng, though many consecutive rows will share the same coordinate.

⚠ GPS Accuracy Note
GPS accuracy on mobile is typically 3–10 meters outdoors. In tunnels, dense urban canyons, or indoors, GPS may be unavailable. The app handles this gracefully — lat/lng columns are left empty when no fix is available, and the map simply stops drawing the polyline until GPS is restored.
Chapter 06

Obstruction Reporting System

Obstructions are semantic events — traffic jams, roadblocks, diversions, accidents — that sensors cannot detect. The rider reports them manually via a dedicated press-and-hold button.

Interaction Flow

  1. Press and hold the ⊘ OBSTRUCTION button
    The button pulses purple. A timer counts the duration of the hold. All sensor samples during the hold are pre-labeled obstruction.

  2. Release the button
    A bottom sheet slides up from the bottom of the screen with four sub-type options.

  3. Tap a sub-type
    Options: 🚗 Traffic Jam · 🚧 Roadblock · 🔀 Diversion · 🚨 Accident. The sub-type is recorded in the sublabel column of the CSV.

  4. Auto-dismiss fallback
    If the rider ignores the sheet for 5 seconds (they may be riding), it auto-dismisses and defaults to "Traffic Jam". The ride continues uninterrupted.

  5. Map marker dropped
    A purple ⊘ marker is placed at the GPS location where the obstruction started. Tapping the marker shows: sub-type, duration, and timestamp.

CSV Output for Obstructions

Obstruction samples appear as regular rows but with label = obstruction and a sub-type in the sublabel column. Non-obstruction samples have sublabel = none.

Chapter 08

Database Schema

SQLite is used for its simplicity — zero configuration, single file, no separate database server to manage. For a data collection project at this scale (thousands of rides), SQLite is more than sufficient.

-- rides table: one row per completed ride
CREATE TABLE rides (
  id            INTEGER  PRIMARY KEY AUTOINCREMENT,
  rider_id      TEXT     NOT NULL,        -- anonymous UUID from phone
  timestamp     TEXT     NOT NULL,        -- ISO 8601 ride start time
  device_model  TEXT,                     -- e.g. "Samsung Galaxy S23"
  os            TEXT,                     -- e.g. "Android 14"
  sample_count  INTEGER  DEFAULT 0,
  window_count  INTEGER  DEFAULT 0,
  ride_duration INTEGER  DEFAULT 0,       -- milliseconds
  label_counts  TEXT,                     -- JSON string
  csv_filename  TEXT     NOT NULL,        -- filename in /uploads/
  uploaded_at   TEXT     DEFAULT (datetime('now'))
);

-- index for fast per-rider queries
CREATE INDEX idx_rider_id ON rides(rider_id);
CREATE INDEX idx_uploaded ON rides(uploaded_at);

Chapter 09

Admin Dashboard

Served at /dashboard, protected by HTTP Basic Auth with a password set in your server's environment variables. It is a single HTML page loaded with live data from your API.

Dashboard Sections

Summary Stats

Total rides collected, total sensor samples, number of unique riders, total ride duration across all rides, and per-label counts as a bar chart.

Rides Table

Paginated table of all rides sorted by upload time. Columns: Rider ID (anonymized), device, OS, samples, duration, label breakdown, upload time. Download CSV button per row.

Global Road Condition Map

A Leaflet map that loads GPS coordinates and labels from all uploaded CSVs and overlays them as colored markers on the map. Over time, this becomes a city-wide road quality heatmap built entirely from your users' rides — your ground truth dataset visualized geographically.

📌 Privacy Note
Rider IDs are anonymous UUIDs generated on the phone. No personal data (name, email, phone number) is ever collected or transmitted. The privacy notice is shown to users on their first ride with a consent checkbox before any upload occurs.

Sample Rate

The devicemotion event fires at approximately 60–100Hz depending on the device and browser. At 100Hz, a 10-minute ride produces roughly 60,000 rows in the CSV. File size is approximately 8–12 MB per 10 minutes uncompressed.

Deployment — Step by Step

📌 Two Services to Deploy
Frontend → Netlify (free, static hosting, instant HTTPS)
Backend → Railway (free tier, Node.js, persistent disk for SQLite)

Part A — Deploy the Frontend (Netlify)

  1. Open Netlify Drop
    Go to app.netlify.com/drop in your browser.

  2. Drag and drop your HTML file
    Drop road_data_collector.html onto the page. Netlify instantly gives you a URL like https://jolly-river-abc123.netlify.app.

  3. Create a free Netlify account
    Claim the site to keep the URL permanently and enable redeployment. Go to app.netlify.com → Sign up → then "Claim site" on your deployed page.

  4. Set your server URL in the HTML file
    Before deploying, edit the one line in the frontend code: const SERVER_URL = 'https://your-railway-app.railway.app' — update this after your backend is deployed (see Part B). Then redeploy by dragging the updated file again.

Part B — Deploy the Backend (Railway)

  1. Create a GitHub repository
    Go to github.com → New repository → name it roadsense-server. Upload your server files: server.js, database.js, middleware.js, dashboard.html, package.json.

  2. Sign up for Railway
    Go to railway.app → Sign up with GitHub (free tier gives you $5/month credit — more than enough).

  3. Create a new project
    Click "New Project" → "Deploy from GitHub repo" → select roadsense-server. Railway auto-detects Node.js and runs npm install && node server.js.

  4. Set environment variables
    In Railway dashboard → your service → "Variables" tab. Add these:

    API_KEY=your-secret-key-here          # for /api/rides dashboard access
DASHBOARD_PASSWORD=your-password       # for /dashboard basic auth
PORT=3000                              # Railway sets this automatically too

  5. Add a volume for persistent storage
    In Railway → your service → "Volumes" → Add Volume. Mount path: /app/uploads. This ensures your CSV files and SQLite database survive server restarts.

  6. Get your Railway URL
    In Railway → your service → "Settings" → "Networking" → "Generate Domain". You'll get a URL like roadsense-server.railway.app. Copy this.

  7. Update the frontend with your server URL
    Edit road_data_collector.html: find const SERVER_URL and replace it with your Railway URL. Redeploy to Netlify.

Part C — Verify Everything Works

  1. Test the backend directly
    Visit https://your-app.railway.app/api/stats in your browser with the API key as a query param: ?key=your-secret-key. You should get a JSON response with zeroed stats.

  2. Open the frontend on your phone
    Open your Netlify URL in Safari (iOS) or Chrome (Android). Grant all permissions. Do a short test ride. End the ride and go through the review screen. Tap "Upload & Download".

  3. Check the dashboard
    Visit https://your-app.railway.app/dashboard. Enter your dashboard password. You should see your test ride appear in the table.

CORS Configuration

Your backend must allow requests from your Netlify domain. In server.js:

const corsOptions = {
  origin: [
    'https://your-app.netlify.app',
    'http://localhost:3000'  // for local testing
  ],
  methods: ['GET', 'POST']
};
app.use(cors(corsOptions));

Chapter 12

Future ML Pipeline

Once you have collected enough labeled rides from real users, the CSV files on your server become a training dataset for a proper machine learning model. Here is what that pipeline looks like:

rides.db + /uploads/*.csv → Data Pipeline Script
↓
Feature Extraction (mean, std, max, jerk per window)
↓
Train Classifier (Random Forest / LSTM)
↓
Trained Model → exported as ONNX / TFLite
↓
Replace Adaptive Engine in frontend with model inference

The adaptive calibration engine is your stepping stone. It makes data collection scalable and produces high-quality labels across diverse riders and vehicles. Once you have sufficient labeled data, a Random Forest or LSTM trained on per-window feature vectors will generalize far better than any threshold-based system.

✓ The Flywheel
More riders → more labeled data → better ML model → replace adaptive engine with ML model → even better labels → even better training data. This is a self-improving data flywheel.
